- **2026-05-22 — `--mode {env,manual}` dataload mode added to `talent_data_pipeline.main`.** Operators can now point a run at a different PG host without editing `app_config/.env`. CLI surface: `--mode env` (default — no prompts, current behavior preserved) or `--mode manual` (interactive prompt at startup). Env-var fallback `DATALOAD_MODE=env|manual` for shell wrappers; precedence is CLI flag > env var > default `env`. Manual mode prompts `PG host [<current PGHOST>]: `, strips whitespace, accepts the default on bare Enter, re-prompts up to 3 attempts when both input AND default are empty, then exits 2. Fails fast with exit 2 + redirect message when `--mode manual` is used without a TTY (so CI never hangs on input). Host override is plumbed through `config.apply_host_override(host)` which mutates `os.environ["PGHOST"]` AND the `db_config` singleton via `object.__setattr__` (bypassing `frozen=True`) — `pg_connect()`'s lazy import and `base_loader._get_pool()`'s lazy construction both pick up the new host at next call. Entra `DefaultAzureCredential` path is fully preserved — only `host` changes; no password fallback, no bypass of the existing config plumbing. Banner line `[pipeline] mode=<mode>  host=<host>  (from .env|overridden via prompt)` prints before `main()` runs. Touched only the inner package (`talent_data_pipeline/talent_data_pipeline/main.py` + `config.py`); per 2026-05-22 cleanup, outer `talent_data_pipeline/{main,config,validate,connectivity_test}.py` stubs are off the runtime path. Verified via 14 isolated helper checks (mode precedence, host-override propagation, prompt behavior) + live `--help` + live no-TTY exit. Decision capsule: `.squad/decisions/inbox/brett-dataload-mode.md`.

- **2026-05-22 — Phase 1 connectivity test bug: PGUSER empty in `.env` → libpq OS-username fallback → `password authentication failed for user "anildwa"`.** Anil reported `uv run --package talent_data_pipeline python -m talent_data_pipeline.main --mode manual` against `tiqpg9a6d3.postgres.database.azure.com` failed at Phase 1 with a password-auth-failed error despite the codebase being Entra-token-only since 2026-05-12. **Hypothesis H1 (Phase 1 bypasses Entra) FALSIFIED:** `main.py` line 14 imports the inner-package `connectivity_test.run_connectivity_test`, which already calls `pg_entra.pg_connect()`; a repo-wide grep across `talent_data_pipeline/talent_data_pipeline/**` found exactly one `psycopg2.connect()` (inside `pg_entra.pg_connect` itself) and zero `PGPASSWORD` references. **Hypothesis H2 (PGUSER misconfigured) CONFIRMED — and more subtle than "short name":** `app_config/.env` line 55 is literally `PGUSER=` (empty value), so `db_config.user` was the empty string, libpq received `user=""`, and libpq's documented behavior is to fall back to the OS account name — which on Anil's Windows box is `anildwa`. The Entra bearer token was issued correctly (for `anildwa@MngEnvMCAP347541.onmicrosoft.com`), but PG looked up the role bound to libpq's `user="anildwa"`, found it didn't match the principal in the token, and rejected with `password authentication failed for user "anildwa"`. **Fix B applied** to `talent_data_pipeline/talent_data_pipeline/pg_entra.py`: new module-level helper `_build_pguser_hint(pg_user, host, libpq_message)` returns a multi-line hint string with three branches (empty PGUSER → OS-fallback hint pointing at `app_config/.env`; PGUSER without `@` → short-name hint; PGUSER with `@` → just host/user context). Both `pg_connect()` and `EntraThreadedConnectionPool._connect()` now wrap their `psycopg2.connect(**kwargs)` call in `try/except psycopg2.OperationalError`; when the message contains `password authentication failed` (case-insensitive), the exception is re-raised as a new `psycopg2.OperationalError(_build_pguser_hint(...))` chained from the original via `raise ... from exc`. All other operational errors propagate untouched. No password fallback. No auto-mutation of PGUSER. No outer-stub edits. py_compile clean.
