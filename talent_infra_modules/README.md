# talent_infra_modules — per-component deployment scripts

> **Status:** Architecture skeleton (2026-05-21). Per-component
> `deploy.ps1` implementations land in subsequent commits — see
> [DEPLOYMENT-ORDER.md](DEPLOYMENT-ORDER.md) for the contract each one
> must satisfy and `<folder>/README.md` for the per-component
> expectations.

This folder contains **five independent PowerShell deployers** for
TalentIQ that target an environment where **most Azure infrastructure
already exists**:

| Pre-existing (NOT created by these scripts)                  |
|--------------------------------------------------------------|
| Resource group                                               |
| Virtual network and subnets (`default`, `pe-subnet`)         |
| Azure Container Apps Environment **(OPTIONAL — deploy `00-container-apps-env/` if missing)** |
| Azure Container Registry                                     |
| Azure AI Foundry account, project, model deployments (gpt-4.1, text-embedding-ada-002) |

Each subfolder deploys exactly **one component**:

| Folder                                                                   | Component                                                                 |
|--------------------------------------------------------------------------|---------------------------------------------------------------------------|
| [`00-container-apps-env/`](00-container-apps-env/README.md)              | **(OPTIONAL)** Azure Container Apps Environment (VNet-integrated) + Log Analytics. Run only when an ACA env does not already exist. Can deploy in **parallel** with `01-postgresql/`. |
| [`01-postgresql/`](01-postgresql/README.md)                              | PostgreSQL Flexible Server with AGE, pgvector, pg_trgm, pg_diskann + Entra UAMI roles |
| [`02-backend/`](02-backend/README.md)                                    | Backend + MCP-sidecar Container App                                       |
| [`03-frontend/`](03-frontend/README.md)                                  | Frontend (React/Vite) Container App                                       |
| [`04-data-loading/`](04-data-loading/README.md)                          | Reference-data + employee-graph + embeddings load                         |

There is no top-level orchestrator. The components are **deployed
independently** in the order documented in
[DEPLOYMENT-ORDER.md](DEPLOYMENT-ORDER.md), and the output of one is the
input to the next.

## Why this exists alongside `talent_infra_v2/`

| Aspect              | `talent_infra_v2/`                                | `talent_infra_modules/`                          |
|---------------------|---------------------------------------------------|--------------------------------------------------|
| Orchestrator        | `azd up` (Bicep + hooks)                          | Direct `az` invocations from PowerShell          |
| Scope               | RG + VNet + ACR + PG + Cosmos + Redis + KV + Foundry + ACA env + 3 container apps + data load | **Just** PG, container apps, data — assumes everything else exists |
| Auth posture        | Full Entra ID (MSAL on UI, JWT on backend, UAMI for PG/Foundry/Cosmos) | **Entra ID auth DISABLED** for backend and frontend; UAMI still used for PG and Foundry control plane |
| Use case            | Greenfield environment, full provisioning         | Partial environment, prove the stack end-to-end on borrowed infrastructure |
| State store         | `.azure/<env>/config.json` + azd vault            | None — every parameter is an env var or prompt   |
| Idempotency         | azd-managed (per-resource Bicep)                  | Each script is responsible for its own idempotency |

**These two folders never share state.** Do not point them at the same
resource group at the same time; in particular, the PostgreSQL server
name generated by `talent_infra_v2/` includes a unique suffix that the
partial-environment scripts do not know.

## Prerequisites

1. **Azure CLI 2.53+** signed in: `az login`
2. **Bicep CLI** (`az bicep install`) — needed by `00-container-apps-env/` and `01-postgresql/`
3. **Docker Desktop** running locally — needed by `02-backend/` and
   `03-frontend/` if you opt out of ACR Tasks
4. **Python 3.11+** with `azure.identity` and `psycopg2-binary` — needed
   only by `04-data-loading/`
5. **Subscription role**: Contributor on the resource group is required.
   Additionally, registering UAMIs as PostgreSQL Entra ServicePrincipal
   admins requires `Microsoft.DBforPostgreSQL/flexibleServers/administrators/write`.
   Deploying a brand-new ACA env via `00-container-apps-env/` also requires
   `Microsoft.Network/virtualNetworks/subnets/write` on the VNet's RG.

## Parameter prompting

Every script accepts parameters in this order (first match wins):

1. **Explicit script argument** (`./deploy.ps1 -ResourceGroup rg-...`).
2. **Environment variable** documented per-script.
3. **Azure discovery picker** for the primary subscription/resource group
  (`az account list`, then `az group list` in the selected subscription).
4. **Interactive prompt** with a sensible default shown in `(default: ...)`.
5. The default itself in non-interactive sessions.

If `-SubscriptionId` / `AZURE_SUBSCRIPTION_ID` is omitted, the script lists
visible subscriptions and lets you select by number, subscription ID, or name.
If `-ResourceGroup` / `AZURE_RESOURCE_GROUP` is omitted, the script lists
resource groups in the selected subscription and lets you select by number or
name. Supplying explicit arguments still bypasses discovery.

Run with `-Force` (or set `$env:CI=1`) to auto-confirm all `y/N`
prompts. See [`shared/README.md`](shared/README.md) for the helper API.

## Authentication is disabled by default

These scripts deploy a **demonstration stack** where the React UI does
**not** require Microsoft sign-in and the FastAPI backend does **not**
validate Entra ID bearer tokens. See [AUTH-DISABLED.md](AUTH-DISABLED.md)
for the exact env-var and build-flag contract, the security implications,
and how to re-enable auth later.

PostgreSQL and Foundry **still use Entra ID** — auth is only disabled
between the user and the application tier. Backend → PG and backend →
Foundry continue to authenticate as the backend Container App's
user-assigned managed identity.

## What this is NOT

- **Not an azd template.** No `azure.yaml`, no hooks, no `.azure/`
  directory, no `azd env` calls.
- **Not a Bicep monolith.** Each component compiles its own (small)
  Bicep file and runs its own `az deployment group create`. Compiled
  Bicep modules from `talent_infra_v2/infra/modules/` may be reused
  as-is when they fit.
- **Not a state store.** There is no record of "the last successful
  deployment". Re-running a script must be idempotent against the
  current state of the resource group; if it isn't, that's a bug.
- **Not a replacement for `talent_infra_v2/`.** Use `v2` for a
  greenfield environment. Use this folder when most of Azure is already
  there and you just want to drop the app on top.

## Deployment lessons baked in

The helpers and per-component contracts encode the operational lessons
from prior TalentIQ deployments:

- **ACA Env ↔ subnet 1:1 mapping** (`ManagedEnvironmentSubnetInUse`). After
  an env is deleted, Azure holds the subnet under a ~30-minute soft-lock
  before another env can claim it. `00-container-apps-env/deploy.ps1`
  pre-checks every env in the subscription before attempting a fresh
  deployment, and fails clearly with the wait-30-minutes guidance when a
  collision is detected.
- **ACA subnet delegation** (`Microsoft.App/environments`). `00-` verifies
  this on pre-existing subnets and refuses to silently mutate the
  delegation. New subnets created by `00-` are always created with the
  correct delegation in place.
- **AGE extension** requires `shared_preload_libraries=age` AND a server
  restart before `CREATE EXTENSION` works. The 01-postgresql script
  must apply both and explicitly restart.
- **UAMI-as-PG-role**: when port 5432 is blocked at the ISP or the
  server is private-endpoint-only, register the UAMI through the Azure
  control plane (`az postgres flexible-server microsoft-entra-admin
  create --type ServicePrincipal`) instead of the SQL
  `pgaadauth_create_principal_with_oid` path. The display name MUST
  match the UAMI's name (PG username = UAMI name).
- **MCP-as-sidecar**: backend and MCP share the **backend** UAMI. PG
  role for the MCP container is `${backendContainerAppName}-identity`,
  not a separate identity. Backend reaches MCP via
  `http://localhost:3002/mcp`.
- **Foundry model deployments serialize**: `dependsOn` is required
  between the project and each `Microsoft.CognitiveServices/accounts/deployments`
  child to avoid `IfMatchPreconditionFailed` ETag races. The scripts in
  this folder assume Foundry already has the deployments (we verify,
  we don't create), so this only matters if the verification fails and
  you fall back to `talent_infra_v2/`.
- **Restart cascade after PG state change**: when an Entra
  administrator is added or `shared_preload_libraries` flips, the
  backend Container App's psycopg2 pool retains stale tokens. The
  02-backend script must offer a `--RestartActive` switch that runs
  `az containerapp revision restart` on the active revision after the
  PG changes are confirmed Ready.
